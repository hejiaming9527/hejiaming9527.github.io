网页数据抓取与可视化实践

发表于 2026/07/18
作者
11 分钟阅读

项目说明

这是一个个人学习实践项目，目标是把公开网页中的结构化信息整理为可分析的数据表，并用图表展示数据分布和变化趋势。项目不抓取需要登录、付费或明确禁止自动访问的页面；实践重点放在完整的数据处理流程，而不是追求采集数量。

项目目的

  • 理解网页请求、HTML 结构与数据提取之间的关系。
  • 使用 Python 将零散页面信息转换为统一的表格数据。
  • 学习检查缺失值、重复值和异常值，避免“图表看起来对，数据其实错”。
  • 使用可视化图表表达数据的分布、排序和时间趋势。
  • 形成可重复运行、便于复盘的数据采集小流程。

项目背景

网页中的公开信息通常适合人阅读，却不一定能直接用于统计。例如，标题、时间、类别和数值可能分布在不同 HTML 标签中，日期格式也可能不统一。这个项目以“采集—清洗—分析—展示”为主线，练习把页面数据转化为有依据的结论。

在实际应用中，数据抓取必须尊重网站的 robots 规则、服务条款、访问频率限制、版权与隐私要求。即使页面能在浏览器中打开，也不代表可以无限量自动请求。

本人参与

本项目为个人学习实践。本人负责需求拆分、网页结构观察、数据字段设计、抓取脚本编写、清洗规则整理、图表制作和结果复盘。

技术方案

环节使用方案作用

数据流程

1
2

公开网页 → HTTP 请求 → HTML 解析 → 字段提取 → DataFrame 清洗
        → 质量检查 → 聚合统计 → 图表输出 → CSV/日志留档

1. 设计字段，而不是先急着抓

在编写代码前，先明确每条记录需要哪些字段。例如：标题、类别、发布日期、来源链接和可统计数值。字段设计决定后续图表是否有意义；如果没有日期字段，就无法可靠地画趋势图。

2. 提取公开页面内容

抓取时应设置合理的请求头、超时和访问间隔。每次请求都要检查 HTTP 状态码：200 表示成功拿到页面，404 表示地址不存在，429 通常表示访问过于频繁。遇到异常应记录原因并停止或退避重试，不要持续高频请求。

1
2
3

response = requests.get(url, headers=headers, timeout=10)
response.raise_for_status()
soup = BeautifulSoup(response.text, "html.parser")

上述代码只展示基本思路。实际项目还应针对目标网站确认其公开访问规则，并限制请求频率。

3. 清洗与质量检查

页面提取出的文本往往包含换行、空格、缺失字段或重复记录。将记录转为 DataFrame 后，先检查数据质量，再统计：

1
2
3
4

df = pd.DataFrame(records)
df = df.drop_duplicates(subset=["url"])
df["published_at"] = pd.to_datetime(df["published_at"], errors="coerce")
df = df.dropna(subset=["title", "published_at"])

errors="coerce" 的含义是：无法识别的日期会变成空值，方便后续集中检查，而不是悄悄产生错误日期。

4. 用图表回答问题

图表前先写清楚问题。例如：

  • 哪些类别的记录数量较多？使用柱状图。
  • 信息发布量是否随时间变化？使用折线图。
  • 各类别在总量中的占比如何？可使用条形图或占比图。

category_count = df.groupby("category").size().sort_values(ascending=False)
category_count.plot(kind="bar", title="各类别记录数量")
plt.xlabel("类别")
plt.ylabel("记录数")

图表只是数据的表达方式，不是结论本身。图表中的时间范围、筛选条件、单位和数据量都应在说明中写清楚。

实现功能

  • 读取目标公开页面并检查响应状态。
  • 从 HTML 中提取标题、类别、时间、链接等预设字段。
  • 将多条记录汇总为 DataFrame。
  • 去除重复链接，处理空值和不规范日期。
  • 按类别和时间进行数量统计。
  • 输出 CSV 数据文件和基础统计图表。
  • 记录失败页面与异常原因，方便再次检查网页结构变化。

遇到的困难与处理方法

网页结构变化导致选择器失效

网页的标签层级和 class 名称可能调整，原来的选择器会提取不到内容。处理时先保存少量 HTML 样本，再通过浏览器开发者工具重新确认结构；代码中对“找不到字段”的情况给出明确日志，而不是直接使用空值继续绘图。

日期和数值格式不统一

同一页面可能同时出现“今天”“几小时前”和标准日期，也可能含有逗号、单位或空格。处理时将原始文本保留一列，再创建清洗后的标准列；不能转换的数据要标记并统计数量，避免默默丢失。

重复与分页数据

分页、置顶内容和页面更新都可能让同一条记录出现多次。项目使用链接或稳定 ID 作为去重依据，并在抓取结束后记录“请求数、解析数、去重后条数”，让数据量变化可解释。

请求频率与网站规则

连续高频请求会给网站带来压力，也容易被限流。实践中采用小规模样本、超时设置和访问间隔；遇到 429、验证码或登录墙时停止自动化请求，改为查阅公开 API 或取得授权。

中文图表显示问题

部分环境中的 Matplotlib 没有可用中文字体，会显示方框。处理方法是在运行环境中选择已安装的中文字体，并在项目说明中注明字体仅影响显示，不改变数据本身。

阶段性工作日志

以下为项目学习阶段的复盘记录，用于展示完整的工程思路；后续应以实际运行日志、代码提交和图表文件作为补充证据。

  1. 需求定义阶段：确定只处理公开页面，梳理目标字段和计划回答的数据问题。
  2. 结构观察阶段：使用开发者工具查看 HTML，记录每个字段所在标签和可能缺失的情况。
  3. 最小抓取阶段：先获取一页数据，验证请求、解析和异常处理流程。
  4. 清洗检查阶段：去重、统一日期格式，统计空值和异常值，确认结果可信。
  5. 可视化阶段：按问题选择柱状图和折线图，为图表补充标题、坐标轴和数据范围说明。
  6. 复盘整理阶段：保存 CSV、图表和运行日志，记录网页变化、限流和数据质量问题。

项目总结

这个项目让我认识到，网页抓取的难点不只是“把内容拿下来”，更重要的是遵守规则、理解数据来源、验证清洗结果，并让图表能够回答明确的问题。后续如果将项目继续完善，可以增加配置文件、单元测试、缓存机制和公开 API 适配；在任何公开展示前，都应删除敏感信息并说明数据来源与采集时间。
